Architecture

Module Overview

Mosaic is organized into four top-level modules with clear boundaries:

ModuleDirectoryPurpose
MosaicKit MosaicKit/ Core logic. Domain models, terminal FFI, IPC, services, AI engine. No UI imports (except AppKit for Ghostty FFI).
MosaicUI MosaicUI/ SwiftUI and AppKit views. App entry point, sidebar, workspace renderer, terminal views, settings.
MosaicGateway MosaicGateway/ Standalone Hummingbird server. Telegram webhook handler, WebSocket bridge to desktop, authentication.
CLI CLI/ Self-contained mosaic command-line tool. Sends JSON-RPC requests to the Unix socket.

Data Flow

User Input (Keyboard)

NSEvent (keyDown)
  --> AppDelegate (three-phase routing)
    --> performKeyEquivalent (shortcuts)
    --> keyDown (normal keys)
    --> interpretKeyEvents (IME)
      --> GhosttyKeyboard (zero-allocation translation)
        --> ghostty_surface_key / ghostty_surface_text (C FFI)
          --> Ghostty renders via Metal

Socket Command

CLI / External Client
  --> Unix Socket (/tmp/mosaic.sock)
    --> SocketServer (accept, read, off-main-thread)
      --> AccessController (auth check)
        --> CommandDispatcher (JSON-RPC 2.0 parse)
          --> CommandRegistry (lookup handler)
            --> *Commands handler (@MainActor for UI mutations)
              --> Manager (WorkspaceManager / PanelManager)
                --> Response (JSON-RPC 2.0)

Mosaic AI

User message (Telegram or ai.message socket command)
  --> MosaicAI
    --> ContextCollector (gather workspace/pane state)
      --> CommandMapper (build system prompt + tools)
        --> LLMProvider (Claude / OpenAI API call)
          --> Parse tool_use response
            --> CommandSafety (validate each command)
              --> CommandRegistry (execute via socket API)
                --> Result relayed back to user

Telegram Alert

Terminal event (process exit, output match, etc.)
  --> EventBus (publish)
    --> AlertEngine (rule-based detection)
      --> GatewayClient (WebSocket)
        --> MosaicGateway (relay)
          --> TelegramChannel (Bot API)
            --> User's Telegram chat

Key Design Decisions

Clean Rewrite

Mosaic is a clean rewrite that references established projects's proven patterns (FFI lifecycle, socket API namespaces, portal pattern, session persistence) but shares no code. established projects had god objects (TerminalController at 15K lines, Workspace at 9.6K, ContentView at 13.3K) and a flat directory structure with 54 files. Mosaic decomposes these into focused types with clear boundaries.

Single Target, Folder-Based Modules

MosaicKit is a folder group within the Xcode project, not a separate Swift package. This keeps the build simple for v1. SPM extraction is planned for pre-v2 when iOS requirements (TerminalBackend protocol, RelayBackend) make the abstraction boundary concrete.

Concrete Types for v1

TerminalSession is a concrete class wrapping Ghostty directly, not a protocol. With only one implementor (Ghostty), designing a protocol with ~30 methods across config, keyboard, display link, search, and copy mode would be premature. Protocol extraction happens pre-v2 when iOS RelayBackend requirements inform the real abstraction boundary.

PaneTree as Indirect Enum

The layout tree uses Swift's indirect enum pattern: 

indirect enum PaneTree {
    case pane(Panel)
    case split(direction: SplitDirection, ratio: Double,
               left: PaneTree, right: PaneTree)
}

PaneTree is immutable. Methods return new trees rather than mutating in place. This makes undo/redo and snapshot persistence straightforward.

Portal Pattern

An AppKit overlay (portal) sits above the SwiftUI view hierarchy for hit-test performance. An isPointerEvent guard separates the keyboard path (fast, zero-allocation) from the mouse path (expensive but infrequent). This preserves the sub-5ms typing latency target.

Three-Phase Keyboard Routing

Keyboard events flow through three phases in the AppDelegate: performKeyEquivalent (system shortcuts), keyDown (normal keys), and interpretKeyEvents (IME input). This matches how macOS processes keyboard input and ensures correct behavior with input methods.

Focus Policy

Only two commands steal focus: workspace.select and pane.focus. All other socket commands preserve the current focus state. This prevents automation scripts and AI commands from disrupting the user's active context.

Threading Model

Socket parsing and JSON validation run off the main thread. @MainActor is applied to managers and command handlers that mutate UI state. Telemetry and event coalescing also happen off-main. The keyboard hot path (GhosttyKeyboard) uses zero allocations.
