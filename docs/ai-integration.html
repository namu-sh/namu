AI Integration

Overview

Mosaic AI is a natural language control plane. It is not an agent bridge, an MCP wrapper, or a chat assistant. When you send a message like "split pane and run npm test", Mosaic AI:

  1. Collects context (workspace list, pane states, working directories, git branches)
  2. Sends the message plus context to an LLM (Claude or OpenAI)
  3. Parses the LLM's tool_use response into structured socket commands
  4. Validates each command through the CommandSafety filter
  5. Executes approved commands via the CommandRegistry
  6. Returns the result to the user

Mosaic AI uses the LLM as a command interpreter, not as an autonomous agent. Every action maps to exactly one socket API command. Batch operations (e.g., "split and run") emit multiple sequential tool_use calls.

LLM Provider Configuration

Mosaic supports swappable LLM backends. Providers conform to the LLMProvider protocol: 

protocol LLMProvider {
    func complete(
        messages: [Message],
        tools: [Tool]
    ) async throws -> Response
}

Supported Providers

ProviderFileAPI
Claude (default) MosaicKit/AI/Providers/ClaudeProvider.swift Anthropic Messages API with tool_use
OpenAI MosaicKit/AI/Providers/OpenAIProvider.swift OpenAI Chat Completions API with function_calling

Configuring a Provider

Provider selection and API key configuration are available in Mosaic's AI preferences (Settings panel). You can also check the current configuration via the CLI: 

mosaic ai status

Returns:

{
    "ready":        true,
    "provider":     "claude",
    "model":        "claude-sonnet-4-20250514",
    "safety_level": "normal",
    "processing":   false
}

Async Execution Model

Mosaic AI follows an async-by-default model:

  • Acknowledge immediately -- "On it..." sent within 500ms
  • Execute asynchronously -- LLM call + command execution
  • Deliver result when ready -- typically within 30 seconds (LLM-dependent)

There is no real-time streaming requirement. Terminal operations are not chat -- a few seconds of latency is acceptable.

Multi-Turn Disambiguation

When intent is ambiguous, Mosaic AI asks clarifying questions. For example: 

User:  "check the logs"
AI:    "You have 3 panes open. Which one has the logs?
         1. Pane in 'Backend' (~/api, running node)
         2. Pane in 'Frontend' (~/web, running vite)
         3. Pane in 'Backend' (~/api, idle)"
User:  "the first one"
AI:    [reads screen from pane 1, returns output]

Conversations are tracked per session via ConversationManager. Sessions expire after 30 minutes of inactivity.

Command Safety

Every command from Mosaic AI passes through CommandSafety before execution. This is mandatory and cannot be bypassed.

Safety Levels

LevelCommandsDescription
safe list, status, read_screen, ping, version, capabilities, get_url, get_title Read-only operations with no side effects
normal create, delete, select, rename, split, close, focus, resize, back, forward, reload, navigate, clear, subscribe, unsubscribe Structural changes to workspaces, panes, or configuration
dangerous send_keys, send_text, send_key, execute_js, message Input injection or process execution that can modify running state

Unknown commands default to dangerous to err on the side of safety.

Behavior by Source

SourceSafeNormalDangerous
Local (socket CLI) Allowed Allowed Allowed (logged)
External (Telegram) Allowed Allowed Requires confirmation

Destructive Pattern Detection

When a send_keys command payload matches a destructive pattern, confirmation is always required regardless of source. Detected patterns include:

  • rm -rf
  • sudo rm, sudo kill
  • mkfs, dd if=
  • chmod 777
  • shutdown, reboot
  • kill -9, pkill -9
  • Fork bombs
  • Pipe-to-shell patterns (curl | sh, wget | sh)

Payloads are normalized (whitespace collapsed, lowercased, escape characters stripped) before matching.

Rate Limiting

External sources (Telegram) are rate-limited to a maximum of 20 commands per minute. Exceeding this limit returns a rejection with the message "Rate limit exceeded".

Logging

Every command is logged with its source (local or external), safety level, and validation result (allowed, requires_confirmation, or rejected).

Alert Engine

The AlertEngine provides rule-based detection via EventBus subscriptions. It does not use the LLM -- alerts are triggered by deterministic pattern matching on terminal events.

Alert Rules

TriggerDetection
Output patterns Terminal output matching error:, FAIL, panic
Exit codes Non-zero process exit codes
Port changes Listening ports appearing or disappearing
Idle duration Shell idle beyond configured threshold (e.g., 30 minutes)

Rules are configurable per workspace.

Alert Flow

Terminal event
  --> EventBus (publish)
    --> AlertEngine (rule match)
      --> GatewayClient (if connected)
        --> MosaicGateway
          --> TelegramChannel
            --> User's Telegram chat

If the gateway is not connected, alerts fall back to in-app notifications only.

Telegram Gateway

MosaicGateway is a standalone Hummingbird server that bridges Mosaic to Telegram. It runs as a separate process -- either locally (localhost:8080 for development) or on a VPS for remote access.

Architecture

Telegram Bot API
  --> MosaicGateway (HTTP webhook: /telegram/webhook)
    --> TelegramChannel (parse update)
      --> SessionManager (find connected desktop)
        --> WebSocket --> GatewayClient (desktop)
          --> MosaicAI (NL processing)
            --> CommandSafety (Telegram = external source)
              --> CommandRegistry (execute)
                --> Result relayed back through chain

Authentication

  1. Desktop generates a 256-bit random pairing token (stored in macOS Keychain)
  2. User links Telegram chat by sending /start to the bot and entering the pairing code in Mosaic settings
  3. Desktop sends the token on every WebSocket connection to the gateway
  4. All Gateway-to-Desktop command messages are signed with HMAC-SHA256 using the pairing token as key
  5. Desktop rejects unsigned or tampered messages

Dangerous Command Confirmation

When CommandSafety blocks a Telegram-originated dangerous command, the gateway sends a Telegram inline keyboard: 

Allow `send_keys "rm -rf /tmp/cache"`?

[Yes]  [No]

The confirmation expires after 60 seconds. Tapping "Yes" executes the command; "No" cancels it.

Setup

  1. Create a Telegram bot via @BotFather
  2. Set the bot token in MosaicGateway configuration
  3. Configure the webhook URL (e.g., https://your-server.com/telegram/webhook)
  4. Start the gateway: swift run MosaicGateway
  5. In Mosaic settings, enter the gateway URL and complete the pairing flow

Gateway Endpoints

EndpointMethodDescription
/telegram/webhookPOSTTelegram Bot API webhook handler

Reconnection

The desktop GatewayClient automatically reconnects with exponential backoff: 1s, 2s, 4s, 8s, up to a maximum of 60s between attempts. An in-app status indicator shows the connection state.

Error Handling

FailureBehavior
LLM API down "AI temporarily unavailable. Terminal and socket API still work." In-app banner displayed.
LLM returns malformed tool_use Raw response logged. Malformed call skipped. User sees "I couldn't understand how to do that. Try rephrasing."
Socket command fails JSON-RPC error returned. MosaicAI reports "That command failed: [error]."
Gateway WebSocket disconnects Auto-reconnect with exponential backoff. Alerts fall back to in-app only.
Telegram rate limited Messages buffered, retried with backoff. User alerted if backlog exceeds 10 messages.

Using AI Locally (Without Telegram)

The ai.* socket namespace enables AI interaction without the Telegram gateway. All AI features are available locally via the CLI: 

# Send a natural language command
mosaic ai message --content "create a new workspace called API"

# Check AI readiness
mosaic ai status

# View conversation history
mosaic ai history --limit 10

Local commands are classified as CommandSource.local, which means dangerous commands are allowed with logging only (no confirmation required).
